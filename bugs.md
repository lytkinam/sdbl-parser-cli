# Известные ошибки работы программы

## Баг с относительными путями `--output-dir`

**Описание:** При использовании относительного пути (`./out`, `out` или просто имя папки) часть артефактов создаётся не в указанной папке, а в **текущей рабочей директории** (корне проекта).

**Как проявляется:**
- `analyze` создаёт `SDBL_PARS/` и `FULL_PARS/` внутри `./out`, но `LINE_PARS/`, `field_lineage/`, `full_field_lineage/`, `RESTORED_QUERIES/` и `VERIFICATION/` — в корне проекта.
- Команда `restore-query` возвращает пустую строку, так как не находит `full_field_lineage` по ожидаемому пути.
- Верификация возвращает пустой отчёт.

**Когда возникает:**
- В Windows при запуске из `cmd` или `Git Bash` с относительным путём.
- На Ubuntu тоже может проявляться при использовании `Path.of("./out")` с `getParent()`.

**Обходное решение:**
Всегда использовать **абсолютный путь** для `--output-dir`:

```bash
# Windows (Git Bash)
bash sd-cli analyze --sql-file query.sql --output-dir "$(pwd)/out" --base-name myquery

# Windows (cmd)
sd-cli.bat analyze --sql-file query.sql --output-dir C:\Users\User\projects\out --base-name myquery
```

**Корневая причина:**
В коде `SdqlPipelineImpl`, `LineParsFieldLineageBuilder` и `QueryReconstructor` используют `outputDir.getParent().resolve("...")`. Для относительного пути `./out` метод `getParent()` возвращает текущую директорию `.`, а не родительскую папку проекта.

## Проблема с кодировкой кириллицы в консоли

**Описание:** При запуске из `Git Bash` или `cmd` русские символы в stderr/stdout отображаются как кракозябры (`��������`).

**Как проявляется:**
- Debug-логи `FFL-DEBUG` содержат нечитаемые символы.
- Имена узлов (например, `Результат_1`) отображаются как `���������_1`.

**Когда возникает:**
- В Windows при выводе в терминал с кодировкой, отличной от UTF-8.

**Обходное решение:**
- JSON-файлы артефактов сохраняются корректно в UTF-8 — читайте результат из файлов, а не из консоли.
- В `cmd` перед запуском выполните: `chcp 65001` (переключение на UTF-8).

## `restore-query` возвращает пустой SQL

**Описание:** Команда `restore-query` может вернуть `"sql": ""` в JSON-ответе.

**Как проявляется:**
- При запросе `restore-query` поле `sql` в ответе пустое.

**Когда возникает:**
- Если `analyze` был запущен с относительным путём (см. первый баг).
- Для простых одноузловых запросов без подзапросов и временных таблиц — возможна особенность реализации.

**Обходное решение:**
- Использовать абсолютный путь при `analyze`.
- Для сложных запросов с временными таблицами восстановление работает стабильнее.

---

## ANTLR `.interp` файлы попадают в git

**Описание:** Сгенерированные ANTLR-артефакты (`*.interp`) меняются при каждой сборке и создают ненужные изменения в `git diff`.

**Статус:** Исправлено — добавлено `*.interp` в `.gitignore`.

---

## Отсутствие готового fat JAR для CLI в стандартной сборке

**Описание:** Стандартная команда `./gradlew classes` не создаёт исполняемый JAR. Для запуска требуется либо bash-скрипт, либо ручная сборка classpath.

**Когда возникает:**
- При попытке запустить утилиту в Windows без Git Bash.

**Обходное решение:**
Использовать задачу `shadowJarCli` (доступна в актуальной версии):
```bash
./gradlew shadowJarCli
java -jar build/libs/bsl-parser-*-cli.jar --help
```
